# Step 1-a: UX シナリオ

## ICAR

- **Intent**: CLI の出力やインタラクションに関わる変更について、Before/After/操作手順を具体的に提示してユーザーと認識合わせする
- **Constraints**:
  - 内部ロジックのみの変更（ユーザー向け出力の変更なし）の場合は「N/A — ユーザー向け変更なし」と明記してスキップ
  - 抽象的な仕様ではなく、**具体的な 1 つの状態**で書く
  - 操作の前後で出力がどう見えるかを明示する
  - 複数シナリオある場合は主要なものを 2〜3 個
  - 提示後はユーザーの承認を待つ。承認が出るまで次のステップに進まない
- **Acceptance**:
  - Before / After / 操作手順 のフォーマットで提示済み、かつユーザーから承認を得ている
  - 承認された UX シナリオは Step 1-c のプランに含めている
- **Relevant**: rules/workflow/1c-plan.md（承認後の組み込み先）

## 提示フォーマット

```
## UX Scenario

### Before
- 具体的な CLI 入出力の状態を記述（どのコマンドで何がどう出るか）

### After
- 同じ操作がどう変わるべきかを記述

### 操作手順
1. ユーザーがどのコマンドを実行したら
2. 出力がどう変わるか
```

提示後、「UX シナリオを確認してください」と明示的に伝え、承認を待つ。
