# Step 1-b: 調査・設計判断

## 1b-A: 調査

- **Intent**: 変更対象とその周辺の現状実装を把握し、設計判断と Refactor Guard の入力を揃える
- **Constraints**:
  - 変更対象から呼び出している・呼び出されている関連コードまで広げて確認する
  - `rules/` のうち関連するものを Read（必要な資料はタスクによる）
  - `references/knowledge.md` を Read してハマりどころがないか確認する
  - 類似実装パターンは Grep で探す
- **Acceptance**: 現状実装・関連コード・既存パターン・既知のハマりどころを把握した状態
- **Relevant**: rules/, references/knowledge.md, rules/architecture.md

## 1b-B: 設計判断

- **Intent**: 調査結果をもとに、モジュール配置・共通化方針を含む主要な設計判断を行う
- **Constraints**:
  - 設計判断の前に `design-principles` スキルを Skill ツールで呼び、返ってきたルールに当てはめる
  - ルールに当てはめても決まらないときだけユーザーに確認する
  - **モジュール配置**: 「依存の方向に違反しないか」「既存モジュールの責務を逸脱しないか」で判断する。新モジュールを切るならその理由を書く
  - **共通化と分離**: 「片方だけ変更したくなったとき、もう片方に影響なく変更できるか？」で判断する。無理に共通化して分岐だらけになるなら分ける
- **Acceptance**: 判断結果がプランに明記された状態（「検討した結果〇〇にした」で十分。該当なしなら省略可）
- **Relevant**: rules/architecture.md, design-principles スキル

## 1b-C: リファクタリングガード

- **Intent**: 変更対象に対して `/refactor-guard` の判定を取り、必要なリファクタを先送りしない
- **Constraints**:
  - 1b-A で特定した変更対象ファイルのパスを引数にして、**メインコンテキストで** `/refactor-guard` を実行する（サブエージェントでは skill を呼べないため）
  - 「判定を無視して続行」は不可。`/refactor-guard` の意義（「できるか」ではなく「やるべきか」を客観判定する）が失われるため
  - backlog の直近バージョンに計画済みのリファクタ指摘は既知として無視してよい。それ以外の新規スメルだけ判断する
  - このステップは必須（省略禁止）
- **Acceptance**: 判定結果と標準対処に従って次の動作（プラン作成 / プランへ含める / 別 plan に切る / backlog 追記 / 中断提示）が決まった状態
- **Relevant**: `~/.claude/skills/refactor-guard/SKILL.md`, `backlog/backlog.md`

### 判定マトリクス（暫定フル記述）

判定は 2 軸の組み合わせで行う:

- **関係性軸**: 今回の変更との関係
  - **関係内**: 今回触っている範囲のリファクタ
  - **範囲外**: 今回触らない範囲で見つかったリファクタ機会
- **規模軸**: リファクタの規模
  - **小規模**: 今回の変更と一緒にやれるサイズ
  - **独立規模**: 別タスクに切るべきサイズ

| 判定 | 標準対処 |
|---|---|
| ✅ 続行可 | そのままプラン作成へ進む |
| ⛔ 関係内・小規模 | 今回の変更と一緒にプランへ含める |
| ⛔ 関係内・独立規模 | リファクタ先行で別 plan に切る。ユーザーに「中断するか」を提示し（規模見積り + 中断時の進め方を併記）、プランモードでは書き込めないため提示のみで停止する |
| ⛔ 範囲外・小規模 | やるなら別 plan、やらないなら backlog に記録。プランに「`backlog/backlog.md` 追加」を 1 ステップとして本来作業の**直後**に置き、本来の作業と一緒にコミットする |
| ⛔ 範囲外・独立規模 | backlog に追加して追跡可能にする。追加方法は上と同じ |

中断提示時に伝える内容: リファクタの規模見積り（変更ファイル数・影響範囲・追加調査の要否）と、中断した場合の進め方（プランモード解除 → `backlog/backlog.md` に本来タスクの**直前**としてリファクタを追加 → コミット → 新規セッションでリファクタ計画から始める）。

backlog 追記はプランモードでは書き込めないため、実装フェーズで実行する作業としてプランに含める。

> **Note**: この判定マトリクスはグローバル `/refactor-guard` の SSoT 化が完了するまでの暫定フル記述。SSoT 化完了後は「Refactor Guard を実行し出力に従う」だけに薄められる。詳細はグローバル `~/.claude/skills/refactor-guard/SKILL.md` 参照。
