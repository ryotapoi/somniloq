# Step 1-b: 調査・設計判断

## 1b-A: 調査

変更対象のコードと関連コードを読み、現在の実装を把握する。

- 変更対象のコードを Read で読み、現在の実装を把握する
- 変更対象から呼び出している・呼び出されている関連コードを確認する
- `rules/` を読む（必要な資料はタスクによって変わるので、関連するものを Read で読む）
- 既存の類似実装パターンを Grep で探す
- `references/knowledge.md` — 関連するハマりどころがないか確認する

不明点があればユーザーに質問する。仮定で進めない。

## 1b-B: 設計判断

設計判断の前に `design-principles` スキルを Skill ツールで呼ぶ。設計時に当てはめる基本ルール（シンプル優先、型優先、DRY、YAGNI、リファクタ先行など）が返ってくるので、以下の判断はそのルールに当てはめながら行う。当てはめても決まらないときだけユーザーに確認する。

調査を踏まえて、以下の設計判断を行う。判断結果はプランに明記する（「検討した結果〇〇にした」で十分。該当しない場合は省略してよい）。

- **モジュール配置**: 新しいコードをどのモジュールに置くか。判断基準は「依存の方向に違反しないか」「既存モジュールの責務を逸脱しないか」。新モジュールを切る場合はその理由を書く。モジュール構成の詳細は `rules/architecture.md` を参照する
- **共通化と分離**: 既存コードとまとめるか、分けるか。「片方だけ変更したくなったとき、もう片方に影響なく変更できるか？」で判断する。無理に共通化して分岐だらけになるなら分ける

## 1b-C: リファクタリングガード

調査で特定した変更対象ファイルのパスを引数にして、**メインコンテキストで** `/refactor-guard` スキルを実行する。サブエージェントでは skill を呼べないため、調査をサブエージェントに委譲した場合もこのステップはメインで行う。

判定結果による分岐:

- **✅ 続行可**: そのままプラン作成に進む
- **⛔ リファクタ先行（今回の変更範囲と直接関係ない）**: 現在のタスクは継続する。プランに「`backlog/backlog.md` にリファクタタスクを追加する」こと自体を 1 ステップとして記載し、本来の作業と一緒にコミットする
  - 追加先は本来の作業の**直後**が基本（状況次第で前後する）
  - プランモードでは書き込みできないため、「backlog 追加」は実装フェーズで実行する作業としてプランに含める
- **⛔ リファクタ先行（今回の変更範囲を修正する前にリファクタすべき）**: ユーザーに「中断するか」を提示する。プランモードでは書き込めないため、提示のみで停止する
  - 提示する内容: リファクタの規模見積り（変更ファイル数・影響範囲・追加調査が必要か）と、中断した場合の進め方（プランモード解除 → `backlog/backlog.md` に本来のタスクの**直前**としてリファクタタスクを追加 → コミット → 新規セッションでリファクタ計画から始める）

「判定を無視して続行」は選択肢に含めない。Refactor Guard の意義（「できるかどうか」ではなく「やるべきかどうか」を客観的に判定する）が失われるため。

backlog の直近バージョンに計画済みのリファクタ指摘は既知として無視してよい。それ以外の新規スメルが見つかった場合のみ判断する。

このステップは省略してはならない。
