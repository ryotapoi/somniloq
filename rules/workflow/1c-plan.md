# Step 1-c: プラン作成・レビュー

## プラン作成

Plan モードでなければ EnterPlanMode する。調査・設計の結果をプランとして整理する。

## 1 プラン 1 コミットの原則

1 プラン = 1 コミットを基本とする。コミット分割が必要なプランは、プラン段階で backlog タスクを分割し、各タスクを別プランとして個別に計画・実装する。

### プラン段階で分割が必要と判明した場合の手順

`review-plan-split` で分割推奨が出た場合（または、プラン作成中に自分で分割が必要と判断した場合）:

1. **ExitPlanMode をユーザーに依頼**: Plan モード中は backlog 等の編集が制限されているため、ユーザーが ExitPlanMode を実行する必要がある
2. **backlog を更新**: 分割後のタスクを backlog に並べる
   - 順番が問題にならないタスク群: backlog のトップレベルに並列で並べる
   - 一連の作業（順番に実施したい）: 親子の階層構造にして、続けてやるべきものが分かるようにする
3. **ユーザーが Plan モードを再有効化**: 1 番目のタスクから再計画する場合、ユーザーが Plan モードに戻す
4. **1 番目のタスクから再計画**: 別プランとして `/plan` から開始する

`/rewind` + summarize で会話を圧縮するか、新規セッションで開始するかはユーザー判断。

## プランレビュー

プランの記述が完了したら、ExitPlanMode より先に必ず `/review-plan-all` スキルを Skill ツールで実行する。レビューを通るまで修正する。

レビュー完了後に ExitPlanMode を呼ぶ。

### レビュー指摘への対応

対応方針を決める前に `design-principles` スキルを Skill ツールで再度呼ぶ。返ってきたルールに当てはめて対応 / 不対応を判断する。当てはめても決まらないときだけユーザーに確認する。

レビュー指摘に対応しない場合、プランを書き直して**考慮したこと**（不要と判断した理由・別タスクに切り出す理由・トレードオフの認識など）が読み取れる形にする。

- 完了宣言（「対処済み」「明記済み」など）だけで済ませず、事実と理由で書く
- 「考慮したこと」は対応する論点の近くに残す。プラン本文を読めば、なぜその論点を採用しなかったかが分かる状態にする

### レビューの収束条件

レビュー指摘が返ってきたとき、以下のどちらかを満たせば収束（次フェーズに進む）。

- 指摘が 0 件
- 残った指摘すべてについて、指摘の根拠（A）が前回の指摘と同じ
  - 「指摘の文面」ではなく「**根拠（why）が同じか**」で判定する
  - 同じ根拠 A の再指摘 = main 側で考慮を書いた論点に対し、レビューが同じ角度から再度指摘している状態。新しい角度ではないので収束扱い
  - 根拠が異なる A' であれば、別の角度として新たに考慮を書き、プランを更新してレビューに戻る

#### 必須停止条件

以下のいずれかに該当した場合、ループを継続せずユーザーに「続行するか / さらに修正するか / 現状で ExitPlanMode するか」を確認する。

- **3 回目以降のレビュー**: 回数ベースの強制停止。判定（A か A' か）の正確性に依存しない。3 回回しても収束 0 件にならない時点で、レビュアーが磨き込みフェーズに入っている可能性が高い
- **A / A' の判定に迷うとき**: 回数に関係なく、判定が厳密にできないなら停止する。「ほぼ同じだけど別の角度かも」レベルの判断が出た時点で確認に切り替える

判定が厳密にできない理由: レビュアー（特に Opus 4.7 系）は同じ根本論点に対して毎周別の角度から指摘を生成する性質がある。main 側で「A 扱い」と判定しても、次の周で同じ論点を別の言葉で再指摘してくるため、A / A' の境界は実運用上判定困難。回数ベースの強制停止で吸収する。
