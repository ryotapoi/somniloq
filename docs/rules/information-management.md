# 情報管理の原則

AI コーディングエージェントを前提とする開発プロジェクトの情報管理原則。
特定のツール（Claude Code, GitHub Copilot 等）に依存しない一般論。

スコープ: 情報の分類・配置・管理の原則。
プロジェクト管理（スケジュール、タスク管理等）は扱わない。

## 前提

- **SSoT（Single Source of Truth）**: 各情報の「正」は1箇所で管理し、他はそこから導出する
- **DRY（Don't Repeat Yourself）**: 同じ情報を複数の場所に書かない

## 権威による配置: docs / llm-wiki

情報を、まず「**矛盾したときどちらが勝つか**」で分ける。区別の根拠は権威であって、書き手が人か AI かではない（rules も decisions も llm-wiki も、人と AI のどちらが書いてもよい）。

- **docs/**: 正本。`プロジェクトが決めたこと`であり、矛盾時に勝つ。ソースから機械的に導出できない。人も AI も責任を持って書き、変更は意図的に行う。ユーザー向けの公開成果物（API リファレンス・CLI マニュアル等）も「ユーザーから見た振る舞い仕様」として docs/specs/ に属する。
- **llm-wiki/**: AI が編纂・保守する作業の入口。正本ではない。docs/ ・コード・テストと**矛盾する**内容は、それらが勝つ。ただし単純な使い捨てキャッシュではない。ソースから再構成した地図と、ソースからは復元できない外部知見（旧 references 相当）の両方を持つ。後者（`regen: none`）はプロジェクト内ソースからは復元できない唯一の補助記録だが、それが設計判断や仕様を**拘束し始めたら** docs/decisions か docs/specs へ昇格させる（llm-wiki/ に正本級の情報を溜めない）。「変更時にどこを読み・何を直すか」をソース全追いより速く掴むための場所。

```text
project-root/
├── docs/            # 正本（プロジェクトが決めたこと。矛盾時に勝つ）
│   ├── rules/       #   方針・スコープ・拘束的制約（must）
│   ├── specs/       #   振る舞い仕様（設計意図込み。公開リファレンスもここ。what）
│   └── decisions/   #   意思決定ログ（why。append-only）
├── llm-wiki/        # AI が編纂・保守する作業の入口（正本ではない。ソースが勝つ）
├── backlog/         # やること（TODO・バグメモ等。追跡可能一時情報）
├── config/          # 実行環境固有の値（.env, CI 設定等）
└── tmp/             # 揮発的一時情報（gitignore 対象）
```

実装コードとテストは、言語・フレームワーク・ビルドシステムの標準配置に従う。汎用原則として `src/` や `tests/` という物理ディレクトリは強制しない。`docs/` は「ドキュメントの正本」を束ねる枝であり、実装コードはコードとして別軸で正本に属する。

`backlog/` と `config/` は docs/ にも llm-wiki/ にも入れない。`backlog/` は外部のタスク管理ツールへ持ち出されうる「やること」であり、ドキュメントの正本とは持ち出しやすさが異なる。`config/` は CI 設定や `.env` のような実行環境固有の値で、ドキュメントの正本とは明確に別種。どちらもルート直下に置く。

### 権威の三層

「矛盾時に勝つか」と「再生成できるか」は独立した 2 軸であり、組み合わせると居場所が決まる。

| 層 | 領域 | 矛盾時 | 再生成 | 性質 |
|---|---|---|---|---|
| **正本** | docs/rules / docs/specs / docs/decisions / ソース / テスト | 勝つ | 不可 | プロジェクトが決めた権威ある情報 |
| **概念・ガイド** | llm-wiki（`regen: compiled`） | 正本/ソースが優先 | 半可 | ソースから統合した理解・ガイド |
| **外部知見** | llm-wiki（`regen: none`） | 昇格で決着（後述） | 不可 | ソースから復元できない唯一の補助記録 |
| **索引・地図** | llm-wiki（`regen: full`） | 正本/ソースが優先 | 可 | ソースから機械的に再抽出できる対応表・地図 |

`regen: none` の「矛盾時」だけは「ソースが勝つ」と言い切れない。プロジェクト内ソースから復元できない唯一の記録なので、ソースとの矛盾では決着せず、設計判断や仕様を拘束し始めたら docs/decisions か docs/specs へ昇格させて決着させる（→ 設計根拠「なぜ外部知見を …」）。

「勝つ・再生成可」の象限は存在しない（再生成できるものが正本を上書きしては困る）。`regen` フロントマターで各ファイルの再生成可否を宣言し、「捨てて作り直せるか」を 1 行で判定できるようにする（→「llm-wiki/」）。

### 導出フロー

```text
プロジェクトの意図（人・AI が定める）
  → docs/rules/            mission, scope, principles, 制約を定める
  → docs/specs/            scope から振る舞い仕様を具体化する（公開リファレンスを含む）
  → project-native tests   specs からテストを書く
  → project-native source  テストと仕様を満たすコードを書く
  → llm-wiki/              正本とソースから理解・地図を編纂し、外部知見をテーマごとに編む（下流・正本に負ける）
```

backlog/ は導出フローのどの段階でも「やること」を溜める場所。config/ は実行環境ごとに独立して決まる。decisions/ は各段階の判断時に横断的に生まれる。llm-wiki/ はフロー全体を横断して編纂・索引するが、いかなる段の上流にもならない。

上流を変えたら下流に影響する。この依存の方向が各領域の関係を決める（作業の順序ではない）。

`docs/specs/` は領域として常設だが、個々のファイルの存在は強制しない。小規模プロジェクトで rules/ とテストの間に明文化すべき仕様がまだなければ、specs/ は空でよい。書くべき振る舞い仕様（ユーザー視点の挙動・互換性・非目標を含む）が現れたら置く。llm-wiki/ も同様に、領域は常設だが中身は任意。

## 情報の分類

### 情報源の定義

情報の種類ごとに「正」の領域を1つだけ定める。

| 情報の種類  | 問い    | 正の所在    |
| ------ | ----- | ------- |
| 原則・制約  | must  | docs/rules/  |
| 振る舞い仕様 | what（設計意図込み） | docs/specs/ |
| 振る舞い検証 | what（自動検証） | プロジェクト標準のテスト配置 |
| 実装     | how   | プロジェクト標準のコード配置 |
| 環境固有の値 | where | config/ |
| 判断理由   | why   | docs/decisions/ |

specs/ とテストは別の問いに答える。specs/ は「何がどう振る舞うべきか、なぜそうすべきか」、テストは「その振る舞いを機械的に検証する」。矛盾した場合は、原則として specs/ を優先してテストを修正する。ただし、現在のユーザー要求・承認済みタスク・rules/・decisions/ と照合し、specs/ が古いと判断できる場合は specs/ を更新する。判断できない場合はユーザーに確認する。

### 正本ではない情報

正本に分類されない情報は領域として llm-wiki/ に置く。これは「1 ファイルに詰め込む」という意味ではない。性格の異なる情報が混在するが領域は 1 つにまとめ、ファイルはテーマ・対象ごとに分け、各ファイルの `regen` フロントマターで再生成可否を宣言する。

**llm-wiki/**（編纂知 + 索引）: AI が編纂・保守する作業の入口。正本ではなく、ソース/正本と矛盾したらそちらが勝つ。中身は性格の異なる情報が混在する:

llm-wiki/ は次の不変条件を満たすために存在する。1 つでも崩れたページは、地図として機能していない（直すか捨てる）:

- **速い**: Coding Agent がそのページを読むことで、関連ソースを全部追うより速く「どこを読み・何を直すか」の当たりがつく。経路・読む場所・注意点が書かれており、ソースを開かないと何も分からない単なる目次の羅列ではない。
- **docs レベルではない**: 正本（rules/specs/decisions）やソースの内容を再掲しない。規範・仕様・設計理由はポインタ（パス・行・ADR 番号）で正本へ送り、ここには「正本を読む前に作業の地図を掴む」ための編纂結果だけを置く。
- **嘘がない**: 行番号・関数名・パスの参照が現在のソースと一致している。ズレたら正本（ソース）を直して再編纂する（地図側を手で書き換えて取り繕わない）。
- **拾える**: index.md から全ページに到達でき、置き場所のルール（特定ソースの罠はコメント / 横断的挙動は地図 / 単一の集約ファイルは作らない）に従っている。


- **索引・地図**（`regen: full`）: ソースから機械的に再抽出できる対応表。コマンド索引、書き込み系の波及地図、型・列挙値の横断対応表、共有 helper の呼び出しサイト対応表など。正本を再掲せずポインタ（ファイルパス・行・ADR 番号）だけを持つ。腐ったら捨てて作り直す。
- **概念・ガイド**（`regen: compiled`）: 仕様・コード・テスト・ADR を読んで統合した理解。「リンク解決とは何か」「新コマンド追加時にどこを読み何を直すか」など。AI が再編纂できるが、出力は毎回同一にはならない。
- **外部知見**（`regen: none`）: プロジェクト内ソースからは復元できない外部由来の知見（旧 references 相当）。外部ライブラリ・言語仕様の罠、エッジケースの実測、回避策の記録。ソースに無い唯一の補助記録。

詳細は後述の「llm-wiki/」。

**一時情報**: 永続させない情報。拘束力があるなら永続領域に移し、ないなら一時情報として扱う。

| 区分 | git 管理 | 例 |
|---|---|---|
| 揮発的一時情報 | gitignore | 作業メモ、実験コード |
| 追跡可能一時情報 | git コミットして削除後も履歴に残す | バックログ |

### 永続領域への記録判断

作業中に得た知見や新たに確定した要求事項は、適切な永続領域に記録する。

判断基準は2つの観点:

- **反復コスト削減**: 記録しないと同じ議論を繰り返す情報
- **破局コスト回避**: セキュリティ・法務・データ損失に関わる情報

判断に迷った場合は記録しない。

## 各領域の詳細

### rules/

目的別にファイルを分割する。以下はファイル構成の例。

- **mission**: プロダクト目的・非目標
- **scope**: 主要機能・ユースケース定義（状態管理は持たない）
- **planning**: 優先度の決め方
- **constraints**: 破ると破局や再議論を招く制約（検出方法併記）
- **principles**: 方向づけのための方針

### specs/

rules/ の概念を具体的な振る舞い仕様に落とし込む場所。テストの上流であり、テストの元ネタとなる。

specs/ はテストとは別の正を持つ。テストが「入力→出力の自動検証」の正であるのに対し、specs/ は「振る舞い仕様とその設計意図」の正。テストだけでは表現できない情報（概念間の関係、設計判断の文脈、シナリオの全体像）を持つ。

ファイル構成の例:

- **domain-model**: ドメインの概念定義、概念間の関係、振る舞いルール（DDD のユビキタス言語に相当）
- **scenarios**: 「こういう場合こうなる」の網羅的な振る舞いシナリオ
- **ux**: 体験設計、画面構造、操作仕様
- **data-model**: データモデル定義
- **overview**: ユーザー視点のコマンド・操作仕様（ux と同性格の「ユーザーから見た振る舞い」）

specs/ は領域として常設だが、個々のファイルの存在は強制しない。小規模プロジェクトで rules/ とテストの間に明文化すべき仕様がまだなければ、specs/ は空でよい。書くべき振る舞い仕様が現れたら置く。

### backlog/

ルート直下に置く（docs/ にも llm-wiki/ にも入れない）。外部のタスク管理ツールへ持ち出されうるため、ドキュメントの正本とは持ち出しやすさが異なる。

導出フローのどの段階でも発生する「やること」を溜める場所。新規の要求、実装中に発見したバグ、改善案など。

拘束力は弱く、テスト・コード・specs/・decisions/ などに吸収されるまでの中間情報。コミット対象とし、完了した項目は整理する。外部ツール（GitHub Issues 等）で置き換えてもよい。

小さいうちは `backlog/backlog.md` のような単一ファイルに保つ。全体の優先順位・重複・マイルストーン境界を判断するため、読める規模なら常に全体を見る。単一ファイルを読むコストが高くなり、作業対象以外の情報がノイズになる場合だけ、`backlog/index.md` とマイルストーン別ファイル（例: `backlog/v0.1.md`, `backlog/later.md`）に分割する。分割後も、各ファイルの先頭にスコープと上位・隣接 backlog への参照を書く。

### decisions/

後から理由を問われうる判断を記録する場所（いわゆる ADR: Architecture Decision Records）。append-only で運用する。

### llm-wiki/

AI が編纂・保守する作業の入口。「変更時にどこを読み・何を直すか」をソース全追いより速く掴むための場所。正本ではなく、ソース/正本と矛盾したらそちらが勝つ。

**`regen` フロントマターで再生成可否を宣言する**。各ファイル先頭で、捨てて作り直せるかを 1 行で判定できるようにする:

```yaml
---
regen: full        # full=機械的に再抽出可 / compiled=AI が読んで再編纂 / none=外部知見・唯一の記録
sources:           # 導出元。regen: none は省略可だが、外部 URL・実測ログ等があれば残す
  - internal/core/build.go
  - docs/specs/overview.md
  - docs/decisions/0004-...
---
```

- `regen: full` — 索引・地図。腐ったら捨てて再抽出する。理想的には生成スクリプトでも吐ける純度。
- `regen: compiled` — 概念・ガイド。ソースを読んで AI が再編纂する。出力は毎回同一にならない。
- `regen: none` — 外部知見。ソースに無い唯一の記録なので作り直せない。手で育てる。

物理フォルダ構成は強制しない。`regen` で性格が宣言されるので、フォルダは読む順序・粒度の都合で後から自由に切ってよい（index.md からリンクで地図を作る）。

**腐敗を防ぐガード**（主に `regen: full` / `compiled` に効く）:

1. **入口判定**: そのファイルの `sources`（導出元）を書けるか確認する。書けない（=どこから来たか言えない）なら、それは正本であって llm-wiki/ ではない。docs/ へ置く。例外は `regen: none` の外部知見で、プロジェクト内ソースから導出できないため `sources` を省略できる。ただし外部 URL・実測ログ・調査メモなど根拠があれば `sources` に残す。
2. **再生成可能性テスト**（`full` / `compiled` のみ）: 削除してソース/正本から作り直したとき、意味的に同一が復元できるか。復元できない固有情報を含むなら、その部分は正本なので docs/ へ移す。
3. **参照ポインタ原則**（`full` / `compiled`）: 正本の内容を再掲しない。パス・行・ADR 番号のポインタだけを持ち、詳細は正本を読ませる。再掲した瞬間に二重管理（DRY 違反）と腐敗が始まる。

**CI でのバイト一致チェックは課さない**。AI による再生成は非決定的で、毎回バイト単位で同じ出力にはならない。「再生成して diff が出たら fail」は守れない空手形なので、整合は上記ガードと人・AI のレビュー、必要なら `sources` パス・最終レビュー日の存在チェック程度で担保する。

**発見の還流**: 編纂中に正本の矛盾・抜け・古さに気づいたら、llm-wiki/ にメモを書き残さない。backlog/（やること）か decisions/（判断）へ還流させ、正本を直してから再編纂する。llm-wiki/ に正本の修正を溜めない。

## 運用ガイド

各領域の定義を補足する実践的な指針。

### rules/

- scope は粒度を荒く保つ（主要ユースケース・主要機能まで）
- constraints は可能なら自動検出できる形にする（linter、CI、テスト等）
- 自動化できない項目にはレビューポイントを併記する
- 検出もレビューもできない項目は decisions/ に移す

### decisions/

- 各記録に日付と文脈を記載する

### specs/

- domain-model はドメインの概念が増えるたびに更新する
- scenarios はテスト追加の元ネタとして使い、テスト完了後も残す
- specs/ とテストが矛盾したら、原則として specs/ を優先する。ただし現在の要求・承認済みタスク・rules/・decisions/ と照合し、specs/ が古い場合は specs/ を更新する

### backlog/

- `backlog/backlog.md` は読める規模なら単一ファイルを優先する
- backlog を分割する場合は `index.md` で全体地図を持ち、各ファイルにスコープと関連ファイルへの参照を書く
- 完了した項目は定期的に整理する

### llm-wiki/

- 各ファイルに `regen`（full / compiled / none）を宣言する。`full` / `compiled` は手で恒久編集しない（直したくなったら正本を直して再編纂）。`none` の外部知見は手で育ててよい
- ページ間リンクはホスティング（GitHub 等）でそのまま辿れる素の Markdown リンク（`[テキスト](相対パス)`）を使う。特定ツール専用の記法に依存しない。リンク整合性チェックにツールを使う場合も、リンク記法そのものはツール非依存に保つ
- `full` / `compiled` は正本を再掲しない。ポインタ（パス・行・ADR 番号）だけを持つ
- 外部・実測・調査由来で正本から導出できない補助知見（ライブラリの罠・エッジケースの実測）は `regen: none` として残す。単一の「知見ファイル」に何でも詰め込まない（雑多な集約ファイルは育つほど読みづらく腐りやすい）。テーマごとにファイルを分け、特定ソースに紐づく罠は下の「そのソースのコメントへ残す」項のとおりコードへ寄せる
- `regen: none` の知見が設計判断や仕様を拘束し始めたら、docs/decisions か docs/specs へ昇格させる（llm-wiki/ に正本級の情報を溜めない）
- 特定のソースを変更するときだけ必要で、横断的な作業入口にならない情報は llm-wiki/ ではなく、そのソースのコメントへ残す
- 編纂中に正本の矛盾・抜けを見つけたら llm-wiki/ にメモせず、backlog/ か decisions/ へ還流させる

## 設計根拠

### なぜ DRY と SSoT か

DRY で重複を排し、SSoT で情報の種類ごとに「正」の置き場所を1つだけ定める。

### なぜ権威で docs/ と llm-wiki/ に分けるか

最上位の区別を「人が書くか AI が書くか」で引くと破綻する。AI も rules や decisions を書き、人も索引を手で直しうるので、書き手では領域が決まらない。決め手は**矛盾したときどちらが勝つか**という権威の差にある。

- プロジェクトが決めたこと（決めごと・仕様・判断理由）は、矛盾したらそれ自体を意図的に直すしかない。これが docs/（正本）。`src/` と紛れる `source/` ではなく `docs/` を名にしたのは、世間一般でパッと意味が通る素直さを優先したため。
- ソース/正本から再構成した理解や、外部由来の補助知見は、矛盾したら正本側を信じて llm-wiki/ を直せばよい。これを 1 つの枝にまとめると、「ここは信じてよい / ここは作業の入口で正本に負ける」の境界が一目で分かり、AI に「ここを読んで育てろ」と指せる入口が 1 つになる。

運用軸に直結するのがこの二本柱の利点: **docs/ = 正本として直す / llm-wiki/ = AI に読ませ育てさせる / tmp/ = 捨てる**。

### なぜ generated/（純キャッシュ専用領域）を作らないか

当初は「機械的に再生成できる索引」を docs/ の外の generated/ に隔離する案だった。しかし、その仕事の行き先は他に全部ある。ユーザー向け公開リファレンス（API・CLI マニュアル）は「ユーザーから見た振る舞い仕様」なので docs/specs/ の正本。内部用の使い捨て索引は llm-wiki/ の `regen: full`。残るのは「人も AI も書かず、コマンド一発で完全自動生成できる成果物」だけで、現実のプロジェクトにそれが現れるのは稀（多くのプロジェクトで「索引・地図」と呼ばれるものすら、AI がソースを読んで編纂した `regen: full` であって機械生成ではない）。領域を 1 つ減らすと読む入口が減り、エージェントが迷わない。本当にコマンド一発生成の成果物が出たら、その時点で任意領域として generated/ を足せばよい。

### なぜ llm-wiki/ をキャッシュと言い切らず「編纂知」とするか

llm-wiki/ の中身は単純な再生成キャッシュではない、という具体的な詰まりからこの位置づけが要る。索引・地図はソースから機械再抽出できる（純キャッシュ）が、概念理解やガイドはソースを読んで統合した編纂物で、毎回バイト一致では再生成できない。さらに外部ライブラリの罠のように「ソースを読んでも復元できない」唯一の記録もある。

これらを「再生成できるか」というフォルダ境界で割ると、器を先に固定することになり中身が育つ前に窮屈になる。代わりに **`regen` フロントマター**（full / compiled / none）で各ファイルが再生成可否を自己申告する形にした。「このファイルは消していいか」が 1 行で判定でき、フォルダ構成は読む順序の都合で後から自由に切れる。

`regen: none`（外部知見）だけは「ソースが勝つ」と言い切れない。プロジェクト内ソースからは復元できない唯一の補助記録なので、ソースとの矛盾では決着しない（外部仕様の変更や実装前提とのズレは起こりうるので「絶対に正しい」わけではない）。これが第二の正本に化けるのを防ぐため、設計判断や仕様を拘束し始めたら docs/decisions か docs/specs へ昇格させる出口を設けている。

### なぜ導出フローで構成するか

SSoT で「正を1箇所にする」としても、開発に必要な情報は複数種類ある。これらをどう整理するかが次の問題になる。

上流を変えたら下流に影響する。この依存の方向がフォルダ構成を決める自然な根拠になる。「情報の種類を列挙してグループ分けする」のではなく、「何が何から生まれるか」を追うことで各領域の境界が決まる。

### 情報源の定義

AI が無限の性能と速度を持つなら、テストとコード（あるいはコードだけ）で十分かもしれない。しかし現実には、開発初期にテストの手がかりとなる上流の情報が必要であり、抽象度の異なる情報は同じものの重複ではなくそれぞれに正がある。

テストで表現できないものがある:

- **目的（なぜ）**: 「ブルートフォース攻撃を緩和するため」という目的はテストに載らない
- **非現実的な検証コスト**: 「GDPR 準拠」をテストだけで表現するのは非現実的
- **優先順位の判断基準**: 「データ整合性を優先する」という方針はテストの記述対象ではない

これらは rules/ や decisions/ に置く。

導出フローの各段階が答える問いに対応させると、複数種類の「正」が自然に現れる。ただし、実装コードとテストの物理配置は言語・フレームワーク・ビルドシステムの慣習に従うため、汎用原則では `src/` や `tests/` というディレクトリ名を強制しない。

### なぜ specs/ をテストと別の正とするか

テストだけでは以下の情報が失われる:

- **設計意図**: テスト名にヒントは書けるが、方針との関連は載らない
- **概念間の関係**: テストが200個あっても全体像は読み取りにくい
- **シナリオの文脈**: テストは入力→出力を示すが「なぜそうすべきか」の背景は示せない

specs/ はテストの上流にある設計文書であり、テストの元ネタとなる。テスト完了後も残す。テストから specs/ を逆生成することは可能だが、コンテキスト消費と安定性の問題がある（llm-wiki/ の「なぜ編纂物をファイルで持つか」と同様の理由）。

specs/ は領域として常設だが個々のファイルの存在は強制しない、としているのは、小規模プロジェクトでは rules/ の概念定義とテストの間に明文化すべきギャップがない場合があるため。領域を常設にするのは、ギャップが生まれたとき「どこに置くか」を毎回考えずに済ませ、置き場所のブレを防ぐため。プロジェクトが成長してテストだけでは設計意図が伝わらなくなったらファイルを足す。

specs/ とテストが矛盾した場合、原則として specs/ を優先してテストを直す。テストは機械的な回帰検出として強いが、古い仕様を固定しているだけの場合もあるため、意図の正にはしない。ただし specs/ も常に最新とは限らない。現在のユーザー要求・承認済みタスク・rules/・decisions/ と照合し、specs/ が古いと判断できる場合は specs/ を更新する。どちらが古いか判断できない場合は、エージェントが推測で決めずユーザーに確認する。

### なぜ編纂物をファイルで持つか（llm-wiki/）

「正本」から導出可能な情報をファイルとして保持することは二重管理に見える。しかし:

- **導出コストの問題**: テストとコードが「正本」であっても、全コマンドの一覧や全体像を毎回導出するのはコンテキスト消費と時間の両面で非現実的。なお、ここでいうテストとコードはプロジェクト標準の配置にある実ファイルを指し、特定の `tests/` や `src/` ディレクトリを前提にしない。
- **非決定性の問題**: 同じ正本から導出しても毎回微妙に表現が変わる。安定した参照先がないと共通理解を持ちにくい
- **参照パターンの問題**: コードレベルの詳細ではなく、全体を俯瞰したい場面がある

二重管理に見えても腐敗しないのは、`regen: full` / `compiled` が正本を再掲せずポインタだけを持ち（参照ポインタ原則）、矛盾時は常に正本が勝って作り直される側だから。CI のバイト一致チェックを課さないのは、AI 再生成が非決定的でその検査が原理的に守れない空手形になるため。整合は入口判定・再生成可能性テスト・参照ポインタのガードと、人・AI のレビューで担保する。

### なぜ外部知見（旧 references）を独立領域にせず llm-wiki/ に吸収するか

導出フローに乗らない情報も現実の開発では必要になる。実装中に発見した技術的知見はテストやコードから読み取れるとは限らず、外部ドキュメントの要点や回避策はプロジェクトの導出フローから生まれたものではない。

これらは「ソースを読んでも復元できない（=再生成不可）」が「プロジェクトが決めた正本そのものでもない」。rules/ に入れると拘束的制約と混ざり、decisions/ に入れると判断記録と事実ベースの知見が混ざる。当初は独立した references/ 領域を立てたが、AI が作業前に読む入口は 1 つにまとめた方が迷わない。そこで llm-wiki/ に `regen: none` として吸収した。地図（`full`）・概念ガイド（`compiled`）・外部知見（`none`）はどれも「作業の入口・正本に負ける」点で共通し、違いは再生成可否だけなので、`regen` フロントマターで表現すれば 1 領域に同居できる。

ただし `regen: none` は唯一の記録ゆえ「正本に負ける」と言い切れない弱さがある。これが設計判断や仕様を拘束し始めたら docs/decisions か docs/specs へ昇格させ、第二の正本に化けるのを防ぐ。

### なぜ scope に実装状態を持たせないか

- 状態管理を持たせると更新頻度が上がり、rules/ の安定性と衝突する
- 実装状態の確認が必要なときは、テスト・コードから AI に判断させる
- 直近でやることは backlog/ で管理する
- scope は「最終的に提供するもの」の一覧として安定させる

### なぜ backlog/ を TODO のみにするか

以前は backlog/ に振る舞い仕様（UX 設計、データモデル等）も置いていたが、これは TODO と仕様の混在を招く。specs/ の新設により、TODO のみを backlog/ に残す形に整理される。

backlog/ は小さいうちは単一ファイルに保つ方がよい。全体を読める規模なら、エージェントは優先順位、重複、マイルストーン境界をまとめて判断できる。早く分割しすぎると、局所ファイルだけを読んで「これは本当にこのバージョンに入れるべきか」「別の項目と統合すべきではないか」を見落としやすくなる。

分割するのは、単一ファイルを読むコストが全体を見ないリスクを上回った場合に限る。その場合も `backlog/index.md` のような全体地図を置き、マイルストーン別ファイルや later ファイルにスコープと関連ファイルへの参照を書く。`active/` のような時間状態名は意味が曖昧になりやすいため、まずは `v0.1.md` などスコープ名で分ける方がよい。

### なぜ一時情報を2種類に分けるか

- **揮発的一時情報**: 作業メモ、実験コード。削除後に履歴を辿る必要がない。gitignore でよい
- **追跡可能一時情報**: バックログ。エージェントがファイルを破損・消失した場合の復元や、要求やタスクの変遷を後から追跡する場面がある

### なぜ記録に迷ったら記録しないか

無条件に記録すると rules/ や decisions/ が肥大化し、参照コストが上がって逆に反復コストが増えるため。
