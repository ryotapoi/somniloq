# Implementation Review — Full Chain

実装レビューの全ステップを順次実行し、指摘の反映ループを回す。
**各ステップは前のステップの完了を待ってから実行すること。同時実行は禁止。**
**明示的に「ユーザーに確認」と記載されたステップ以外は、ユーザー確認なしで次のステップへ自動的に進む。**

ユーザーが codex スキップを指示している場合、手順 9-10 をスキップする。

## 手順

### 0. ビルドとテストを通す

```bash
go build ./...
go test ./...
```

失敗したら修正してから次へ進む。

ユーザー動作確認は `/verify-user-check` によって `2b-verify.md` の段階で既に通過している前提。このスキルでは実行しない（本手順 11 で再度実行する）。

### 1. diff サイズを確認する

`git diff --stat` で変更行数を確認する。合計が 1000 行を超える場合、AskUserQuestion で「diff が大きいため Codex レビュー（/review-code-codex）でレート制限に当たる可能性があります。Codex レビューも実行しますか？」と確認する。ユーザーが No なら手順 9（/review-code-codex）をスキップして手順 11 に進む。

### 2. プランファイルの更新（必要な場合のみ）

プランから意図的に変更した箇所がある場合、プランファイルを更新する（該当 Step に変更内容と理由を追記）。

### 3. `/simplify` を Skill tool で実行する

結果を簡潔に報告したら、ユーザー確認を待たずに次の手順へ自動的に進む。

### 4. `/review-code` を Skill tool で実行する

引数（`$ARGUMENTS`）があればそのまま渡す。

### 5. `/review-code-go` を Skill tool で実行する

引数があればそのまま渡す。

### 6. `/review-code-somniloq` を Skill tool で実行する

引数があればそのまま渡す。

### 7. 新規の 🔴 MUST / 🟡 SHOULD 指摘を実装に反映する

判断が必要な指摘は AskUserQuestion でユーザーに確認する。収束判定は `rules/workflow/2c-review.md` の「レビューの収束条件」に従う。

### 8. 新規指摘があった場合 → ループ

- 手動確認が必要な変更を行った場合（CLI 出力・対話的挙動・実 DB/ログへの副作用に影響する修正等）→ 手順 0 に戻る（ビルド・テスト → 手動確認再判定 → レビュー）
- 自動テストのみで十分な変更の場合 → 手順 4 に戻る

新規 MUST/SHOULD がゼロになるまで codex 以外のレビュー群（手順 4〜6）でループする。

### 9. 手順 1 で Codex スキップが決定している場合、手順 11 に進む

### 10. `/review-code-codex` を Skill tool で実行する

**引数は自然言語で渡すこと**（harness 制約により `--resume` や `--flag` 形式のフラグは `$ARGUMENTS` 全体を空にしてしまう）:

- 初回（プランあり）: `args: "プランファイル <PLAN_PATH> に対する実装をレビューしてください"`
- 初回（プランなし）: `args: "プランなしで git diff をレビューしてください"`
- 2 回目以降: `args: "<上記初回文> を前回の続きで再レビューしてください"`

Codex の出力に 🔴 MUST / 🟡 SHOULD / 🔵 NIT の指摘がある場合（LGTM でない場合）、`/codex-findings-append` スキルを Skill tool で実行して追記する。引数: `code somniloq "<変更概要>"`

### 11. 新規指摘があれば反映し、手順 8 と同じルールでループ先を判断する

Codex の指摘を反映した場合、整合性を取るためにもう一度 codex 以外のレビュー群を全部走らせる。

Codex レビューが 2 回目の場合、自動で反映せず指摘内容をユーザーに提示する。ユーザーがさらにループするか終了するか判断する。

### 12. ユーザー動作確認の再判定

レビュー対応でコードを変更した場合、`/verify-user-check` を Skill tool で実行する。判定が「要確認」なら CLI 出力・対話的挙動・実 DB/ログへの副作用などを実際に動かして確認し、必要ならユーザーに依頼する。通過してから次へ進む。

変更なし、または自動テストで十分にカバーできる変更のみの場合はスキップしてよい。
