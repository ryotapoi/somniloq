# Implementation Review — Risk-Based

実装レビューを、作業のリスクに応じた深さで実行する。
**準備手順と並列レビューは順次実行。同一手順内の個別レビューは並列起動する。**
**明示的に「ユーザーに確認」と記載されたステップ以外は、ユーザー確認なしで自動進行する。**

ユーザーが codex スキップを指示している、または手順 0 の diff サイズ確認でユーザーが Codex No と答えた場合、`/review-code-codex` を並列リストから除外する。

## Review Depth の選択

最初に main 側で diff の risk profile を判定し、Depth を選ぶ。判定が揺れたら一段深い側に倒す。

- **L0 self-check**: Small 変更（typo、docs、テスト追加だけ、1 ファイルの明確な修正）。main で diff を読み直し、acceptance と照合するだけ。手順 0 のビルド・テストを通したら手順 7 (commit) に直接進む
- **L1 targeted**: 通常の変更。手順 0〜2 + 領域固有レビューを実施
- **L2 external**: High-risk、設計判断が重い、曖昧、または L1 で重大な指摘が出た。L1 に加えて Codex を入れる

判定例:
- typo / docs / test 追加だけ / 1 ファイルの明確なバグ修正 → L0
- 通常の機能追加・バグ修正・複数ファイル変更 → L1
- SQLite スキーマ・マイグレーション・`backfill` 破壊的処理・SQL 集約意味変更・CLI 破壊的変更・JSONL 取り込み境界・並行性・公開 API 削除・外部連携 → L2

## 用語

以下の用語を使う（個別レビュースキルが返す `needs_action=YES/NO` の言い換え）:

- **指摘あり観点**: 前周で `needs_action=YES`（1 件以上の MUST/SHOULD 指摘）を返した観点
- **LGTM 観点**: 前周で `needs_action=NO`（指摘ゼロ）を返した観点

`/simplify` は従来形式（戻り値本文のみ）で結果を返すため、本文に MUST/SHOULD の指摘が含まれていれば「指摘あり」、なければ「LGTM」として扱う。**`/simplify` はチェーン全体で 1 周目のみ実施し、再実施対象には含めない**（レビュー + 修正の混合スキルのため）。

## 手順

### 0. 前提整備

#### 0-a. ビルドとテスト

```bash
go build ./...
go test ./...
```

失敗したら修正してから次へ進む。

CLI 動作確認は `.claude/workflow/verify.md` の段階で既に通過している前提。このスキルでは実行しない（最後の手順でレビュー対応の差分について再判定する）。

#### 0-b. diff サイズ確認（L2 のみ）

`git diff --stat` で変更行数を確認する。合計が 1000 行を超える場合、`AskUserQuestion` で「diff が大きいため Codex レビュー（/review-code-codex）でレート制限に当たる可能性があります。Codex レビューも実行しますか？」と確認する。ユーザーが No なら以降の手順 1〜N の並列起動から `/review-code-codex` を除外する。

#### 0-c. プランファイルの更新（必要な場合のみ）

プランから意図的に変更した箇所がある場合、プランファイルを更新する（該当 Step に変更内容と理由を追記）。

### 1. リスクに応じた観点を並列で実施する（1 周目）

Depth と diff の内容に応じて並列起動するスキルを選ぶ。**選んだものを 1 つのメッセージで Skill tool 並列起動** する。引数（`$ARGUMENTS`）はそれぞれにそのまま渡す。

| スキル | いつ含めるか | 出力形式 | 備考 |
| --- | --- | --- | --- |
| `/simplify` | L1 / L2 で常時 | 従来形式 | **1 周目のみ実施。再実施対象から除外。戻り値受領後はユーザー確認なしで手順 4 へ進む（止まりやすい）** |
| `/review-code` | L1 / L2 で常時 | 結果ファイル化 | |
| `/review-code-somniloq` | L1 / L2 で常時（somniloq 制約は常に確認） | 結果ファイル化 | |
| `/review-code-go` | Go コードに触る変更（実質常時） | 結果ファイル化 | |
| `/review-code-codex` | L2 のみ（codex スキップ指示 / 0-b で No 回答の場合は除外） | 結果ファイル化 | |

各スキルの戻り値テキストから:
- 結果ファイル化対象: `^RESULT_FILE: ` 行と `^SUMMARY: ` 行を抽出する
  - `RESULT_FILE:` の値が `ERROR` で始まる場合、本文がそのまま戻り値内に含まれているのでフォールバックとして扱う（戻り値本文を読む）
- 従来形式（simplify）: 戻り値本文をそのまま指摘として扱う

`/review-code-codex` の戻り値テキスト 1 行目は `plan mode: ...` のカナリア。続く行から `^RESULT_FILE: ` / `^SUMMARY: ` を抽出する。

### 2. 1 周目の結果を main で統合し、まとめて反映する

**指摘反映の進め方**:

1. 全スキル実行が完了するまで結果ファイルは Read しない（戻り値の `RESULT_FILE` / `SUMMARY` 行のみ受け取る）
2. 全スキル完了後、結果ファイル化対象のうち `SUMMARY: ... needs_action=YES ...` のものについて、`RESULT_FILE` のパスを Read で読み込む
   - パスが `/tmp/claude/claude-review-results/` 配下であることを確認してから Read する（パス検証）
   - `needs_action=NO` のスキルの結果ファイルは Read しない
3. 従来形式スキル（simplify）の本文と合わせて全指摘を一覧し、🔴 MUST / 🟡 SHOULD 指摘の対応方針を決定する。対応方針の判断と収束条件は呼び出し元 workflow（`.claude/workflow/review.md`）に従う
4. 対応方針が決まったら Edit に入る。隣接セクションへの修正は 1 つの Edit にまとめる。離れたセクションへの修正は別 Edit のまま
5. 反映完了後、結果ファイルは再 Read しない

**main の責務**: A/A' 判定（同根拠の磨き込みかどうか）は main では行わない。main は受け取った指摘ごとに「採用 / 却下」だけ判断する。A/A' 判定は再レビュー時に fork 側で行う（手順 4 参照）。

判断が必要な指摘は `AskUserQuestion` でユーザーに確認する。

`/review-code-codex` が `SUMMARY: ... needs_action=YES ...` を返した場合、結果ファイルを Read した上で `/codex-findings-append` を実行する。引数: `code somniloq "<変更概要>"`。`needs_action=NO` の場合は呼ばない。

### 3. ループ前のハードリミット判定（再実施ループに入る前に必ず実行）

これが何周目のレビューかを main 側で数える（手順 1 が 1 周目、再実施が 2 周目以降）。

3 周目以降に入ろうとしている場合、ユーザーに状況を簡潔に報告して指示を仰ぐ:

- 周回数
- 残っている指摘の要旨
- 同根拠の指摘（前周以前で skip した指摘と why が同じ）が残っているか

報告は状況説明にとどめ、選択肢は提示しない。ユーザーの指示があるまでループに戻らない。

### 4. 部分再実施ループ（2 周目以降）

前周で「指摘あり観点」だった個別レビューのみを **同一ターンで並列起動** する。LGTM 観点はスキップ。`/simplify` は再実施対象に**含めない**（1 周目限定）。

**手動確認が必要な変更を入れた場合**（CLI 出力・対話的挙動・実 DB/ログへの副作用に影響する修正等）→ 並列起動の前にビルド・テスト（手順 0-a）を再度通す。自動テストのみで十分な変更なら省略してよい。

**再レビュー時の引数渡し**: 自然言語で前回指摘 + main 側の対処を渡す（codex 系の従来パターンに揃える）:

プランあり:

```text
プランファイル <PLAN_PATH> に対する実装を前回の続きで再レビューしてください。
前回の指摘への対処:
- 🔴 <指摘1の要旨> → <対応内容を1行 or 対応しない理由を1行>
- 🟡 <指摘2の要旨> → ...
新規問題と、対処の妥当性を確認してください。
```

プランなし:

```text
プランなしで git diff を前回の続きで再レビューしてください。
前回の指摘への対処:
- 🔴 <指摘1の要旨> → <対応内容を1行 or 対応しない理由を1行>
- 🟡 <指摘2の要旨> → ...
新規問題と、対処の妥当性を確認してください。
```

引数本文に「Claude 側の見解」を混ぜず、事実（何を変えたか・どの指摘を残したか）だけ渡す。

`/review-code-codex` の再実施では、前回 Codex レビューの結果ファイルを Read で読んでから引数を組み立てる。**ファイルパスは前回 `/review-code-codex` の戻り値テキストから抽出した `RESULT_FILE:` の値をそのまま使う**（main の会話履歴に残っている）。タイムスタンプや glob で「直近のもの」を選ぶのは禁止 — 他セッションが並行稼働している時に他人のファイルを掴む。`/tmp/claude/claude-review-results/` 配下であることを確認する。

戻りを全部受け取ったら手順 2 と同じ要領で main で統合・反映する。再実施でも `/review-code-codex` が `needs_action=YES` を返したら `/codex-findings-append` を呼ぶ。

全観点 LGTM になるまで手順 3 → 4 をループする。3 周目に入る前は必ず手順 3 のハードリミット判定を通す。

### 5. 最終念押し判定（全観点 LGTM 達成時）

全観点が LGTM になった時点で、main は **直近周までの累積修正** を見て「方向性が変わる規模」かを判定する。

直近周の修正規模で最終念押しの要否を決める。

- **大（最終 1 周回す）**: 設計の方向反転、実装の構造変更、新しい型・状態の導入、複数モジュールにまたがる書き直し
- **中（最終 1 周回す）**: 既存関数のロジック追加、テストケース追加、エラー処理の段落単位追加
- **小（回さない）**: typo、コメント追加、変数名の言い換え、1 行レベルの修正

中と小の境界で迷ったら中扱い（最終 1 周回す）に倒す。

「最終 1 周回す」と判定した場合、**直近周で回さなかった観点**（=前周で LGTM だった観点）を 1 度だけ並列で回す。`/simplify` は対象外。引数は手順 4 と同じ自然言語形式。新規指摘が出たら手順 2 と同じ要領で main で反映し、その指摘あり観点だけをもう一度回す（このとき手順 3 の周回カウントは継続する）。

### 6. CLI 動作確認の再判定

レビュー対応で CLI 出力・対話的挙動・実 DB/ログへの副作用に影響しうる変更を入れた場合、実際に `bin/somniloq <args>` を叩いて挙動を確認する（`2b-verify.md` の観点）。必要ならユーザーにコマンド実行と出力共有を依頼する。

自動テストで十分にカバーできる変更のみ、または変更なしの場合はスキップしてよい。

### 7. `/commit` を Skill tool で実行する
